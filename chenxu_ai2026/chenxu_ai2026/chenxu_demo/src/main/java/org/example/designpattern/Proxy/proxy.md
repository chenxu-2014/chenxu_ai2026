### 1. 代理模式简介
**代理模式**是一种结构型设计模式，它为另一个对象提供一个替身或占位符以控制对这个对象的访问。

**核心思想**：通过引入一个代理对象，在客户端和目标对象之间起到中介作用，从而在不改变目标对象代码的情况下，增加额外的功能（如权限检查、日志记录、事务管理、延迟初始化等）。

---

### 2. 静态代理
静态代理在编译期就已经确定代理对象和被代理对象的关系。我们需要手动为每一个需要代理的类创建一个代理类。
**静态代理的缺点：**

+ **冗余**：如果接口有很多方法，代理类需要实现所有方法，即使增强逻辑相同，也会产生大量重复代码。
+ **不易维护**：接口一旦变更，真实对象和所有代理类都需要修改。
+ **灵活性差**：代理类和目标类是编译期静态绑定的，一个代理类只能代理一种类型的对象。

---

### 3. 动态代理
动态代理在运行时动态生成代理类，无需手动编写代理类的源代码。Java 主要提供了两种方式：**JDK 动态代理**和 **CGLib 动态代理**。

### 3.1 JDK 动态代理（基于接口）
JDK 动态代理是 Java 标准库提供的功能，位于 `**java.lang.reflect**` 包中。它**要求目标对象必须实现至少一个接口**。

**核心类：**

+ `**java.lang.reflect.Proxy**`：用于生成代理类实例。
+ `**java.lang.reflect.InvocationHandler**`：调用处理器接口，开发者需要实现它来自定义增强逻辑。


_注意：_`_**$Proxy0**_`_ __是 JDK 运行时动态生成的类。_

---

### 3.2 CGLib 动态代理（基于子类）
CGLib (Code Generation Library) 是一个强大的、高性能的代码生成库。它**通过继承目标类，生成其子类来创建代理**。因此，它不需要目标对象实现接口，可以代理普通类。


_注意：_`_**$$EnhancerByCGLIB$$...**_`_ __是 CGLib 运行时动态生成的子类。_`_**finalMethod**_`_ __没有被增强，因为 final 方法不能被重写。_

---

### 4. JDK 代理 vs CGLib 代理 对比总结
| **特性** | **JDK 动态代理** | **CGLib 动态代理** |
| --- | --- | --- |
| **机制** | 基于**接口**，通过反射生成一个实现了相同接口的代理类。 | 基于**继承**，通过字节码技术生成目标类的子类。 |
| **要求** | **目标对象必须实现接口**。 | 目标类**不能是 final 的**（否则无法继承）。 |
| **性能** | **在 JDK 8+ 后性能相差无几**。早期版本中，JDK 代理生成代理对象较快，但调用方法较慢（反射）。 | 早期版本中，CGLib 生成代理对象较慢，但调用方法较快（使用 FastClass 机制直接调用）。**现代 JVM 对反射优化很好，差距已不明显。** |
| **局限性** | 只能代理接口中定义的方法。 | 无法代理 **final 类** 和 **final 方法**、**private 方法**。 |
| **依赖** | Java 标准库自带，无需额外依赖。 | 需要引入 **CGLib 库**。 |


**如何选择？**

+ 如果目标对象**实现了接口**，默认使用 **JDK 动态代理**（如 Spring 的默认选择）。
+ 如果目标对象**没有实现接口**，则必须使用 **CGLib 动态代理**。
+ 从设计角度看，优先使用接口编程，因此 JDK 代理是更自然的选择。但在需要代理第三方库的类时，CGLib 必不可少。

---

### 5. 在源码中的应用
**1. Spring Framework (AOP)**  
Spring AOP 是代理模式最经典的应用。它同时使用了 JDK 动态代理和 CGLib 动态代理。

+ 如果一个 Bean **实现了接口**，Spring 默认使用 **JDK 动态代理**。
+ 如果一个 Bean **没有实现接口**，Spring 会自动使用 **CGLib 动态代理**。
+ 你也可以通过配置 `**@EnableAspectJAutoProxy(proxyTargetClass = true)**` 强制 Spring 对所有 Bean 都使用 CGLib 代理。

**2. Spring 事务管理 (**`**@Transactional**`**)**  
声明式事务的本质就是通过 AOP 代理实现的。在方法调用前开启事务，在方法执行后提交或回滚事务。

**3. MyBatis (**`**Mapper**`** ****接口)**  
MyBatis 的 `**Mapper**` 接口我们只需要定义，不需要写实现类。MyBatis 在运行时使用 **JDK 动态代理** 为这些接口生成代理对象。当我们调用 `**mapper.selectUser()**` 时，代理对象会拦截方法调用，并将其转换为执行 `**SQL**` 语句等底层操作。

**4. RPC 框架 (如 Dubbo, gRPC)**  
客户端调用远程服务接口时，框架会通过**动态代理**（通常是 JDK）生成一个本地代理。调用代理方法时，代理会负责将方法名、参数等信息序列化并网络传输到远程服务器，等待结果返回，从而实现像调用本地方法一样调用远程服务。

**5. Hibernate (延迟加载 Lazy Loading)**  
Hibernate 为了实现延迟加载，返回的实体对象实际上是一个 CGLib 增强的代理对象。当你第一次访问关联对象（如 `**user.getOrders()**`）的属性时，代理对象才会去数据库真正加载数据。
