```java
package org.example.thread_cx;

public class DeadlockDemo {
    private static final Object resource1 = new Object();
    private static final Object resource2 = new Object();

    public static void main(String[] args) throws InterruptedException {
        Thread threadA = new Thread(() -> {
            synchronized (resource1) {
                System.out.println("Thread A acquired resource1");
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                synchronized (resource2) {
                    System.out.println("Thread A acquired resource2");
                }
            }
        });

        Thread threadB = new Thread(() -> {
            synchronized (resource2) {
                System.out.println("Thread B acquired resource2");
                try {
                    Thread.sleep(100);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                synchronized (resource1) {
                    System.out.println("Thread B acquired resource1");
                }
            }
        });
        threadA.start();
        threadB.start();
    }

}
```

### 方案一：破坏“循环等待”条件（最推荐，改动最小）
**核心思想**：让两个线程**以完全相同的顺序**获取锁。只要获取顺序一致，循环依赖链就被打破了。

```java
// 只需要修改 Thread B 的加锁顺序，改为先获取 resource1，再获取 resource2
Thread threadB = new Thread(() -> {
    synchronized (resource1) {   // 改这里：先拿 resource1
        System.out.println("Thread B acquired resource1");
        try {
            Thread.sleep(100);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        synchronized (resource2) { // 再拿 resource2
            System.out.println("Thread B acquired resource2");
        }
    }
});
```

### 方案二：破坏“持有并等待”条件（使用 `ReentrantLock` 尝试一次性获取）
**核心思想**：线程在获取 `resource1` 时，如果发现 `resource2` 也被占用，就**立即释放已经占有的 **`**resource1**`，并重试。确保线程不会死死攥着 `resource1` 去等 `resource2`。

```java
import java.util.concurrent.locks.ReentrantLock;

public class DeadlockDemo {
    private static final ReentrantLock lock1 = new ReentrantLock();
    private static final ReentrantLock lock2 = new ReentrantLock();

    public static void main(String[] args) {
        Thread threadA = new Thread(() -> {
            while (true) {
                if (lock1.tryLock()) { // 尝试获取锁1
                    try {
                        if (lock2.tryLock()) { // 尝试获取锁2，如果拿不到立即返回false，不阻塞
                            try {
                                System.out.println("Thread A acquired both locks");
                                break; // 成功获取，跳出循环
                            } finally {
                                lock2.unlock();
                            }
                        }
                    } finally {
                        lock1.unlock(); // 如果没拿到锁2，立刻释放锁1，不持有任何锁去等待
                    }
                }
                // 轻微休眠避免活锁（一直空转）
                try { Thread.sleep(10); } catch (InterruptedException e) {}
            }
        });
        // Thread B 同理（略，代码逻辑一致）
        threadA.start();
    }
}
```

### 方案三：破坏“不可抢占”条件（使用 `ReentrantLock` 的超时机制）
**核心思想**：给获取 `resource2` 的操作设置一个**超时时间**。如果超过时间没拿到，就主动释放 `resource1`，并稍后重试。这相当于让锁变得“可抢占”。

```java
import java.util.concurrent.TimeUnit;
import java.util.concurrent.locks.ReentrantLock;

public class DeadlockDemo {
    private static final ReentrantLock lock1 = new ReentrantLock();
    private static final ReentrantLock lock2 = new ReentrantLock();

    public static void main(String[] args) {
        Thread threadA = new Thread(() -> {
            while (true) {
                lock1.lock(); // 拿到锁1
                try {
                    System.out.println("Thread A got lock1, trying lock2...");
                    // 尝试等待500ms获取锁2，超时则放弃
                    if (lock2.tryLock(500, TimeUnit.MILLISECONDS)) {
                        try {
                            System.out.println("Thread A got lock2, done!");
                            break;
                        } finally {
                            lock2.unlock();
                        }
                    } else {
                        System.out.println("Thread A timed out, releasing lock1 and retry");
                        // 超时未拿到锁2，主动放弃锁1（相当于被“抢占”了）
                    }
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    lock1.unlock(); // 释放锁1
                }
                try { Thread.sleep(100); } catch (InterruptedException e) {}
            }
        });
        // Thread B 同样逻辑（略）
        threadA.start();
    }
}
```

### 方案四：破坏“互斥”条件（使用读写锁的共享模式）
**核心思想**：如果业务逻辑允许**同时读取**资源（比如本例只是打印日志，并未修改 `resource1` 和 `resource2` 的状态），我们可以放弃 `synchronized` 的排他锁，改用 `ReentrantReadWriteLock` 的**读锁（共享锁）**。既然资源可以共享，就不存在互斥，自然就没有死锁。

```java
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class DeadlockDemo {
    // 使用读写锁替代 Object
    private static final ReentrantReadWriteLock rwLock1 = new ReentrantReadWriteLock();
    private static final ReentrantReadWriteLock rwLock2 = new ReentrantReadWriteLock();

    public static void main(String[] args) {
        Thread threadA = new Thread(() -> {
            // 两个线程都只加读锁（共享锁），可以同时进入，不会互斥
            rwLock1.readLock().lock();
            try {
                System.out.println("Thread A acquired resource1 (read lock)");
                rwLock2.readLock().lock();
                try {
                    System.out.println("Thread A acquired resource2 (read lock)");
                } finally {
                    rwLock2.readLock().unlock();
                }
            } finally {
                rwLock1.readLock().unlock();
            }
        });

        Thread threadB = new Thread(() -> {
            rwLock2.readLock().lock(); // 这里即使是先拿2再拿1，因为都是读锁，互相不阻塞，不会死锁
            try {
                System.out.println("Thread B acquired resource2 (read lock)");
                rwLock1.readLock().lock();
                try {
                    System.out.println("Thread B acquired resource1 (read lock)");
                } finally {
                    rwLock1.readLock().unlock();
                }
            } finally {
                rwLock2.readLock().unlock();
            }
        });
        threadA.start();
        threadB.start();
    }
}
```

### 总结对照表
| 解决方案 | 破坏的条件 | 实现手段 | 适用场景 |
| --- | --- | --- | --- |
| **方案一** | ④ 循环等待 | 统一加锁顺序（A和B都先拿R1再拿R2） | **生产环境最常用**，代码侵入小，性能最好 |
| **方案二** | ② 持有并等待 | `ReentrantLock.tryLock()`<br/> 尝试拿全部，拿不到就全释放 | 适用于锁粒度较细，需要极高灵活性的场景 |
| **方案三** | ③ 不可抢占 | `tryLock(timeout)`<br/> 超时自动释放已占有锁 | 适用于必须防止线程长期阻塞的弱一致性业务 |
| **方案四** | ① 互斥 | 改用 `ReadWriteLock`<br/> 的读锁，允许多线程共享资源 | **局限性最大**，仅适用于纯读操作或无需修改共享变量的场景 |


如果拿不准用哪个，**首推方案一**，因为它是通过改变编码习惯（固定锁顺序）来解决，既不引入复杂API，也没有额外性能开销。其他三种方案虽然更“炫技”，但通常只在无法修改锁顺序（比如调用第三方库）时才作为备选。
